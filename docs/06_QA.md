# 06 — QA

QA verifies that an issue's **acceptance criteria** are actually met — the way a careful human tester
would. Human-like, visual verification is the default; headless tests are the fallback; impossible
tests are deferred, never blocking (P7, FR-16–19).

## Philosophy

> QA behaves like a real human: launch the application, navigate, click buttons, fill forms, pair
> devices, take screenshots, compare, read logs, repeat until success.

The QA worker supplies the *judgment* ("does this screen meet the criterion?"); the deterministic
toolbelt supplies the *hands* (launch, tap, type, screenshot, diff, OCR, log-read). Every action is
a zero-token tool call; only the pass/fail judgment spends tokens.

## The QA ladder (pick the highest rung that's possible)

1. **Human-like visual QA (preferred).** Drive the real app on a real device/simulator: launch,
   navigate to the feature, perform the user actions, capture screenshots, image-diff against golden
   baselines, OCR text to assert content, read runtime logs for errors. Judge each AC item.
2. **Headless/automated tests.** When visual is impossible (no UI, pure API/lib, no device), run the
   project's automated tests / API assertions via the toolbelt.
3. **Static verification.** When neither runs (e.g. hardware firmware with no board), run the
   deterministic simulators/validators the plugin provides (build, DRC/ERC, Wokwi/Renode sim,
   FCL-fit) — see [10_Hardware](10_Hardware.md).
4. **Defer.** If none of the above can execute for an environmental reason, **DEFER** (below).

The QA worker records which rung it used and why, as evidence.

## Deterministic QA tools (zero-token)

Provided by the core + plugins ([11_Plugins](11_Plugins.md)):

- **Launch/drive:** `app.launch`, `app.stop`, `ui.tap`, `ui.type`, `ui.swipe`, `ui.wait_for`
  (backed by adb / simctl / Appium-WDA / browser automation per plugin).
- **Capture:** `screenshot.capture`, `screen.record`.
- **Compare:** `imgdiff.compare(golden, actual)` → similarity score + diff image; `ocr.read(region)`
  → text.
- **Logs:** `log.tail`, `log.grep`, `log.since` (device logs, app logs, Sentry snapshot).
- **Devices:** `device.list`, `device.pair`, `device.connected?` (real phone preferred per owner
  rule C-4).
- **Headless:** `test.run(suite)`, `api.request`, `api.assert`.

The QA worker calls these; it never scripts them from scratch and never spends tokens executing them.

## App/screen maps & golden images (Brain)

- The Brain holds a **navigation map** (screens, how to reach them) and **golden screenshots**
  (baseline per screen/state). Maps are built deterministically by an app-crawler tool and refined
  as features change.
- QA navigates using the map, captures the current screen, and `imgdiff` against the golden. New/
  changed screens update the golden **only** after the AC is confirmed met (so goldens track intended
  UI, not regressions).
- This makes most visual QA a **deterministic compare** with a **single reasoning judgment** at the
  end ("does the diff represent the intended change and meet the AC?").

## Verdicts

QA returns one of three verdicts (schema in [05_Workers](05_Workers.md)):

- **PASS** — every runnable AC item verified with evidence. → Merge.
- **FAIL(reasons)** — one or more AC items not met; each failure is **specific and actionable**
  (criterion, what was observed, evidence id). → back to Developer (attempt++).
- **DEFER(reasons)** — one or more checks cannot run for an environmental reason. → record deferral,
  create follow-up issue, treat as non-blocking, merge the rest (FR-18).

Mixed cases: if some AC items PASS and others must DEFER (but none FAIL), the issue may merge with
recorded deferrals. If any item FAILs, the issue loops regardless of deferrals.

## Deferral rules (never block — P7)

A check is **deferrable** only for environmental reasons, e.g.:

- **Hardware unavailable** — the PCB/board/device isn't attached.
- **No device connected** — no phone/simulator available for a UI check.
- **Visual QA impossible** — no rendering path in the current environment.
- **Human-only QA** — needs a human action the engine can't perform (e.g. physical bench test,
  BS1363 plug insertion, notarized store review).

Deferring records: kind, reason, exact how-to for later, the environment required, and a linked
follow-up Jira issue. Branches are **never** held open waiting; the dashboard tracks open deferrals
so the owner clears them when the environment is ready. A deferral is **not** a pass — it's an
honest "not verified yet, here's how", and it is visible.

## Anti-false-green (owner rule: green check = all OK)

- A PASS requires **evidence** per AC item (screenshot/log/test id). No evidence → not a PASS.
- DEFER is never rendered as a green pass; it's a distinct, visible state with a follow-up issue.
- Simulated/static verification is honestly labeled as such (esp. hardware — a sim validates
  *function*, never replaces the legally-required physical bench test; see
  [10_Hardware](10_Hardware.md)). The engine never shows a false green light.

## Flakiness handling (deterministic)

- Known-flaky signatures live in the Brain (known failures). The build/test gate retries a flaky
  suite a bounded number of times deterministically before counting it as a real failure, and
  records flakiness for the owner.
- Image-diff uses tolerance thresholds (config per plugin) to avoid failing on trivial rendering
  noise, while still catching real visual regressions.

## Portability

QA strategy is data + plugin tools: which rung applies, which launch/drive backend, tolerance
thresholds, and where goldens live — all from config/plugin ([13_Config](13_Config.md)). The QA
worker's charter and the verdict schema are project-agnostic (P10).
